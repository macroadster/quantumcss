Can a Blog Template Run on Almost Pure HTML?

This page is a proof of concept for template-scoped semantic styling. The root class chooses the visual system, while raw HTML elements inherit layout, hierarchy, and atmosphere with minimal author input.

The Constraint

Utility CSS works, but it bloats HTML. If a template can infer the difference between navigation, article body, supporting panels, and footer chrome, we can preserve expressiveness while cutting token cost.

The tradeoff is explicitness. A semantic template only succeeds if its assumptions are narrow, stable, and documented.

What the Template Infers

  • The first top-level `nav` becomes a sticky glass header.
  • The main region becomes a two-column editorial shell.
  • The article header gets hero typography and lede spacing.
  • Sidebar sections become compact panels without extra classes.
The real test is not whether this looks good. It is whether the visual language stays predictable once the markup stops being hand-tuned.

Where It Starts to Need Hooks

Hero sections, promoted cards, media galleries, and mixed commerce blocks still benefit from explicit roles. Semantic defaults get you most of the way, but not all the way. 

<html class="blog">
  <body>
    <nav>...</nav>
    <main>
      <article>...</article>
      <aside>...</aside>
    </main>
    <footer>...</footer>
  </body>
</html>
